# ADR 0001: Layered Architecture with Service-Layer Entry Point

**Status:** Accepted  
**Date:** 2026-07-06

---

## Context

Wobsongo is a Go backend. As the codebase grows, we need a clear, enforceable rule about what belongs where — otherwise business logic, transaction management, and database access tend to scatter across layers, making the system harder to test, understand, and maintain.

---

## Decision

We adopt a strict layered architecture with one primary rule:

> **Any part of the system that needs to interact with data or trigger business logic uses the service layer. The service layer is the single entry point for all business logic.**

Repos are private implementation details of services. Handlers, workers, and CLI commands never touch repos directly.

### The Layers

```
┌─────────────────────────────────────────┐
│  HTTP Handlers  (internal/handler/)     │  ← transport only
├─────────────────────────────────────────┤
│  Services       (internal/service/)     │  ← ALL business logic lives here
├─────────────────────────────────────────┤
│  Repos          (internal/repo/)        │  ← private to services
├─────────────────────────────────────────┤
│  DB             (internal/db/)          │  ← SQLC-generated, never hand-edited
└─────────────────────────────────────────┘

Cross-cutting:
  internal/data/   ← repo + provider interfaces (the contracts)
  internal/model/  ← domain models
  internal/queue/  ← job definitions and JobEnqueuer interface
```

### Rules per Layer

**Handler** (`internal/handler/`)
- Parse and validate the incoming request (DTO binding + `go-playground/validator`).
- Call exactly one service method.
- Format and write the response.
- No business logic. No direct repo access. No transactions.

**Service** (`internal/service/`)
- Owns all business logic.
- Is the only layer that depends on repo interfaces (`internal/data/`).
- Orchestrates transactions: when multiple SQL operations must be atomic, the service calls `repo.WithTx(ctx, func(txRepo) error { ... })` and composes individual atomic repo calls inside.
- Enqueues background jobs via `repo.Enqueue()`, optionally inside `WithTx()` for atomic write + enqueue.

**Repo** (`internal/repo/`)
- Implements interfaces defined in `internal/data/`.
- Each method performs **exactly one SQL statement**. No repo method calls another repo method. No repo method manages its own transaction.
- Maps database errors to application-level errors (e.g., `pgx.ErrNoRows` → `ErrNotFound`).
- Maps `db.*` types (SQLC output) to `model.*` types.

**DB** (`internal/db/`)
- SQLC-generated code from SQL files in `sql/queries/`.
- Never edited by hand.

**Workers** (`internal/worker/`)
- Implement `river.Worker[T]`.
- Call service methods for all business logic — never touch repos or DB directly.

**CLI commands** (`cmd/`)
- Use service methods for any data operations — no direct DB connections in command handlers.

### Transaction Pattern

```go
// ✅ Service composes atomic repo methods in a transaction
func (s *DocumentService) Ingest(ctx context.Context, form *dto.IngestDocumentDTO) error {
    return s.repo.WithTx(ctx, func(txRepo data.DocumentRepoer) error {
        if err := txRepo.Create(ctx, doc); err != nil {
            return err
        }
        if err := txRepo.Enqueue(ctx, &queue.ProcessDocumentDTO{ID: doc.ID}); err != nil {
            return err
        }
        return nil
    })
}

// ❌ Repo manages its own transaction — NOT allowed
func (r *DocumentRepo) IngestWithSetup(ctx context.Context, ...) error {
    return pgx.BeginFunc(ctx, r.pool, func(tx pgx.Tx) error {
        // multiple SQL ops...
    })
}
```

### Interface Composition for Atomic DB + Enqueue

When a repo needs to support both transactional DB operations and job enqueuing in the same transaction, its interface embeds both:

```go
type DocumentRepoer interface {
    Crudable[model.Document]
    data.TxAware[DocumentRepoer]
    queue.JobEnqueuer
}
```

### Mock Convention

Mocks are generated by [`moq`](https://github.com/matryer/moq) and live in `internal/mockrepo/`. Each interface in `internal/data/` should have a corresponding mock. Mocks are triggered by `//go:generate` directives in the interface source file.

---

## Anti-Patterns to Avoid

| Anti-pattern | Why it's wrong |
|---|---|
| Repo method calls `pgx.BeginFunc` or `pool.Begin` | Transactions belong at the service layer |
| Repo method does more than one SQL statement | Breaks testability; obscures business logic |
| Worker opens its own DB connection | Bypasses service layer; breaks observability |
| CLI command calls `db.Queries` directly | Same as above |
| Service imports `internal/db` directly | Services should depend on repo interfaces, not SQLC output |

---

## Consequences

- Services become a readable, testable boundary: all business logic is visible in one place.
- Each repo method is trivially mockable (one SQL op = one mock call).
- The rule is verifiable in code review: does this repo method contain more than one `q.` call? Does this worker import `internal/repo`?
- New features follow a predictable checklist: model → SQL query → repo method → mock → service → handler.
